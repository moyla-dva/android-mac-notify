# Notification Dispatch Strategy

## 1. Product Intent

通知分发的目标不是把 Android 通知完整搬到 Mac。当前收敛后的默认策略是：

```text
通知只自动接力验证码和链接。
文件通过用户主动分享接力。
其他通知默认不跨设备。
```

系统仍保留两步判断：

1. Android 端先判断这条手机事件是否值得跨设备。
2. Mac 端再判断通过门禁的事件应该落到哪个处理表面。

Mac 处理表面包括：

- 菜单栏弹窗：有可执行动作时主动打开，用户处理后自动收起
- 动作收件箱：主窗口内保留尚未处理的动作
- 最近历史：可回看，但不主动打扰
- 跳过：空通知、重复事件或不该持久化的敏感内容

系统通知不再作为普通接力事件的默认入口，只保留给动作失败等需要额外提示的异常反馈。

当前 Phase 1 先使用 Android 端内置硬规则，后续再考虑动态评分和用户反馈。

更上层的事件路由模型见 [Event Routing And Notification Interaction Spec](./EVENT-ROUTING-SPEC.md)。本文件只描述通知分发策略，不能替代事件到动作的整体路由模型。

## 2. Android Relay Eligibility

Android 端第一道门只判断“是否值得穿过设备边界”，不负责完整产品分发。

默认发送：

- 验证码、动态码、安全码、登录码
- 通知正文或标题里的可打开链接
- 用户主动分享给本 App 的文件

条件发送：

- 目前不做默认条件发送；后续如果加入用户自定义，再开放“指定 App / 联系人 / 关键词”。

默认不发送：

- 普通微信、QQ、Telegram、钉钉、飞书聊天通知
- 文件传输助手、Saved Messages、发给自己的普通文本通知
- 外卖、快递、门禁、取件、订单过程等弱确定性状态
- 普通支付成功、普通登录成功、普通安全提示
- 电商营销、内容推荐、频道流、音乐推荐、会员权益、流量包
- 系统常驻、播放、下载、上传、同步、电量和 VPN 状态

这一步的目标是只让确定性强、能在 Mac 上执行的事件进入 Mac，而不是把手机生活搬到 Mac。

## 3. What Must Be Preserved

这些通知默认应该保留，因为它们能在 Mac 上直接完成动作：

- 验证码、动态码、安全码、登录码
- 可打开链接，包括会议链接、文档链接、网页链接

其他“看起来重要但不能在 Mac 上确认处理”的通知，默认不跨设备。

## 4. What Should Be Quiet By Default

这些通知默认不跨设备；如果已经进入 Mac，则只入历史，不主动打开菜单栏：

- 电商营销：优惠、福利、红包、秒杀、金币、会员、抽奖、促销
- 金融营销：理财、贷款、额度、保险、收益、借呗
- 内容推荐：小红书新动态、B 站关注更新、音乐新歌上线、特别企划、开播、推荐内容
- 频道流：Telegram 资源频道、下载地址、破解/辅助工具、标签化长内容
- 系统推荐：应用市场游戏、任务中心、会员权益、流量包
- 助手推荐：房价、房源、商品或服务推荐

普通微信、QQ、TIM、Telegram、钉钉、飞书不再全量跨设备。它们只在含验证码或链接时发送。

未知 App 不再保守发送；没有验证码或链接就不跨设备。

## 5. Hard Rules vs Dynamic Selection

Phase 1 使用硬规则处理安全边界：

- Android 前置门禁跳过空通知、常驻状态、普通 IM、高置信营销和系统状态噪音
- 没有验证码或链接的通知默认不发送到 Mac
- 验证码不写入长期历史
- 重复 eventId 去重
- 低价值无动作通知不跨设备
- 已识别动作的通知优先展示动作
- 内容流和营销来源即使带链接，默认也不跨设备

后续再加动态选择：

- 按应用、联系人、关键词、时间、Mac 活跃状态打分
- 根据用户点击、忽略、复制、打开、清除等反馈调整权重
- 场景模式只改变分发强度，不改变事件模型

原则：

```text
硬规则保护底线。
动态评分决定打扰强度。
用户反馈负责个性化。
```

## 6. Current V1 Implementation

当前代码已实现：

- Android 端 `AndroidNotificationRelayGate` 在读配置、入队和发 HTTP 前先做跨设备门禁
- 空通知、常驻状态、普通 IM、高置信营销、系统播放/下载/电量等默认不发送到 Mac
- 验证码和通知链接默认发送
- 地址、自我接力普通文本、外卖/会议状态、交易异常、未知普通通知不再默认发送
- 通过 Android 门禁的微信、QQ、短信类通知只有验证码或链接会进入 Mac
- Mac 端不再做第二套营销/静默规则，默认信任 Android 门禁结果
- 通过门禁的事件进入 Mac 菜单栏和动作收件箱，而不是默认发 macOS 系统通知
- 动作执行结果会按 actionId 记录，成功动作从待处理区退出并跨重启保留
- 动作结果持久化只保存成功元数据，不保存执行反馈文案或通知正文
- 文件投递使用独立文件卡片，不混入普通通知收件箱和最近历史
- 外卖、会议等状态卡片 provider 当前默认关闭，只作为后续显式配置扩展

重要 reason code：

- `verification_code`
- `open_link`
- `shared_file_received`
- `actionable_notification`
- `transient_sensitive`

## 7. Known Limits

当前仍是保守规则，不是完整智能分发：

- Android 端前置门禁还没有用户可配置界面
- Android 端还没有“最近被拦截”诊断页，用户暂时无法直接看到某条通知为什么没有接力
- 不能识别具体联系人或群聊优先级
- 不能判断同一 App 内不同频道的用户偏好
- 营销词仍可能有少量误伤，需要后续用户反馈或诊断层修正
- 状态卡片 provider 代码仍保留，但默认不从 Android 通知主链触发
- “发送 / 不发送”的强度还没有用户可配置界面
- 当前规则粒度还没有扩展到联系人、群聊、频道或场景

下一步如果继续做通知体验，优先补：

1. 为联系人、群聊、频道增加更细粒度的匹配键
2. Android 端增加“最近拦截 / 放行原因”诊断视图
3. 会议、来电等新的显式事件 provider
4. 外部物流事件接入规范，用于未来接收快递短信解析、行业 API 或其他项目输出的标准物流事件
5. 基于 Mac 活跃状态的分发强度调整
