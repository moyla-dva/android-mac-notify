# Android Mac Notify MVP Spec

## 1. 项目定位

`android-mac-notify` 是一个面向 `Android + Mac` 的本地协同工具。

第一阶段不做通用设备协同平台，也不做全功能 KDE Connect 替代，而是聚焦一条高价值主线：

- 把 Android 手机上的重要事件带到 Mac 上
- 让用户在 Mac 上直接完成最关键的处理动作

MVP 的核心价值不是“同步一切”，而是：

- 重要通知到达 Mac
- 验证码在 Mac 上一键复制
- 链接在 Mac 上一键打开
- 在不同本地网络环境下尽量稳定可恢复

## 2. 目标用户

- 主力手机是 Android
- 主力工作设备是 Mac
- 工作时不想频繁拿起手机
- 经常需要处理验证码、重要通知、链接
- 可能在不同网络环境下使用：
  - 家庭 Wi-Fi
  - 办公 Wi-Fi
  - Android 手机热点

## 3. MVP 一句话定义

一个在不同本地网络环境下，稳定把 Android 重要通知和验证码带到 Mac 上处理的本地协同工具。

## 4. 核心用户问题

当前用户痛点：

- 在 Mac 上工作时，仍然需要频繁拿手机看通知
- 登录网站或系统时，需要手动去手机上取验证码
- 手机上看到重要链接，切到 Mac 继续处理不顺滑
- 网络环境变化后，跨端工具容易直接失效

MVP 要解决的最小问题集：

- 重要通知能到达 Mac
- 验证码能在 Mac 快速复制
- 链接能在 Mac 快速打开
- 连接能在普通 Wi-Fi 和手机热点下工作
- 出问题时用户能知道原因

## 5. 默认主路径

用户默认路径：

1. 第一次打开 Mac 端，看到配对入口
2. Android 端输入或扫码导入 Mac 地址与 token
3. 建立一对一连接
4. Android 收到重要通知
5. Mac 弹出本地通知
6. 如果识别到验证码，用户一键复制
7. 如果识别到链接，用户一键打开
8. 网络切换后自动重连，失败时给出明确提示

## 6. MVP 功能范围

### 6.1 P0 基础内核

必须实现：

- Android 与 Mac 一对一设备绑定
- 设备 ID 与本地信任关系
- token 或密钥机制
- 稳定会话
- 心跳或状态同步
- 自动重连
- 连接状态可见
- 本地日志
- 最近成功连接信息记忆

### 6.2 P1 网络环境能力

MVP 必须考虑不同网络环境，不允许只在“理想同 Wi-Fi 环境”下成立。

必须支持：

- 普通同一 Wi-Fi 下连接
- Android 手机热点下连接
- 自动发现作为增强能力
- 手动输入地址作为兜底能力
- 扫码导入地址和 token
- 发现失败时仍可手动直连
- 网络切换后自动重试
- IP 变化后的恢复连接

至少需要区分这些状态：

- 未配对
- 正在连接
- 已连接
- 网络不可达
- 认证失败
- 已断开，等待重连

### 6.3 P2 主功能

MVP 主功能只围绕通知、验证码、链接展开。

Android 端：

- 使用 `NotificationListenerService` 监听通知
- 抓取最小通知字段：
  - `appPackage`
  - `appName`
  - `title`
  - `text`
  - `postedAt`
  - `notificationKey`
- 基础过滤：
  - 忽略空通知
  - 忽略明显低价值常驻通知
  - 支持应用白名单
- 将通知发送到 Mac

Mac 端：

- 运行本地接收服务
- 常驻菜单栏应用
- 收到通知后弹出 macOS 通知
- 保存最近通知历史
- 显示连接状态与当前设备

动作能力：

- 自动提取验证码
- 一键复制验证码
- 自动识别链接
- 一键在 Mac 打开链接
- 基础去重，避免重复轰炸

### 6.4 P3 轻量主动接力

MVP 可以包含一个很轻的主动接力能力，但范围必须收住：

- Android 主动发送链接到 Mac
- Android 主动发送文本到 Mac

这部分只作为 MVP 的补充，不应反过来拖慢通知主链。

## 7. 明确不做

MVP 阶段不做：

- iPhone 支持
- 文件传输
- 双向剪贴板
- 多设备支持
- 查找手机
- 短信会话
- 通用自动化规则平台
- 云端转发
- 远程控制
- 全功能设备协同平台

## 8. UX 原则

- 默认路径只有一条，先配对再使用
- 自动发现不是唯一入口
- 用户必须看得懂当前连接状态
- 失败时必须有明确反馈
- Mac 是主处理台，Android 端尽量轻
- 规则先少而有效，不先做复杂配置系统

## 9. 成功标准

MVP 达标条件：

- 普通 Wi-Fi 下首次配对顺利完成
- Android 热点环境下也能完成连接
- 重要通知能稳定到达 Mac
- 验证码复制链路顺滑
- 链接打开链路顺滑
- 网络切换后能恢复，或至少能明确诊断
- 日常使用时通知噪音可控

## 10. 主要风险

- Android 厂商后台策略可能影响长期稳定性
- 热点环境下自动发现不一定稳定
- 通知筛选策略过弱会导致噪音过大
- 过早加入文件传输等能力会让项目失焦

## 11. MVP 之后的优先扩展

完成 MVP 后，优先考虑：

1. Android 主动分享链接到 Mac
2. Android 主动分享文本到 Mac
3. 文件快速投递到 Mac
4. 查找手机
5. 多设备支持
