# UI Interaction Direction

## Product Role

Mac 端不是通知列表，也不是开发诊断台。

它的产品角色是：

```text
Android event action inbox on Mac
```

用户平时不应该频繁打开主窗口。最高频路径应发生在 Mac 菜单栏弹窗和主窗口动作收件箱。

## Target User

- 主力手机是 Android
- 主力工作设备是 Mac
- 需要在桌前快速处理验证码、链接和文件投递
- 不想被所有手机通知打断

## Core Task

用户真正要完成的是：

- 复制验证码
- 打开链接
- 接收手机侧文件
- 知道连接是否可靠

不是：

- 读取 Host / Port / Token
- 管理原始通知流
- 观察调试字段

## Default Path

```text
首次配对 -> 菜单栏弹窗处理 -> 主窗口回看和管理 -> 二级可靠性/诊断入口
```

界面职责：

- Android 接力页：查看接力是否正常、暂停/恢复接力、打开可靠性检查
- Android 分享页：显示当前文件投递任务和修复入口
- Android 设备页：管理附近 Mac、本机名称和手动连接兜底
- 菜单栏弹窗：即时动作、快速状态和最近动作
- 主窗口：动作收件箱、进行中状态、最近记录
- 可靠性页：通知访问、系统通知和后台放行

## Android IA

Android 端不是控制台，也不是配置页。

默认路径：

```text
打开 App -> 看协同状态 -> 处理待修复能力 -> 选择附近的 Mac -> Mac 上确认 -> 打开通知访问
```

界面规则：

- 首页第一屏必须表达 Android 与 Mac 的连接关系，而不是表单。
- 通知接力、文件投递等能力以状态块展示，让用户知道哪些能力已经可用。
- 已配对设备必须提供“暂停接力 / 恢复接力”，用于临时停止通知和文件发送。
- “暂停接力 / 恢复接力”必须同步给 Mac；Mac 端应显示“接力已暂停”，不能继续误报“已连接运行中”。
- “忘记这台 Mac”是更重的动作，会清掉手机端本地配对并回到重新连接流程。
- 首页不直接暴露 Host、Port、Token。
- 可靠性不占底部主 Tab。它是接力页右上角的二级入口，并由首页的问题卡在需要时主动提示。
- 附近设备是主入口，手动地址只作为“没有找到 Mac？”的折叠入口。
- 本机名称是辅助设置，命名为“这台手机”，不叫“设备显示名”。
- 从系统分享进入时，直接进入文件投递页，不先展示配对表单。
- 文件投递失败或未配对时，才展开连接修复入口。
- 错误文案必须解释用户下一步，不暴露协议字段。

视觉方向：

- 不使用 Android 模板默认紫色，也不依赖 Material You 动态取色。
- 固定使用“设备蓝 + 接力绿 + 提醒橙”的产品色，让连接、可用、待处理有稳定语义。
- 视觉重点放在设备连接关系和能力状态，不做电商式高饱和促销视觉。
- 首页主视觉使用深蓝设备桥接面板，手机和 Mac 是视觉主角。
- 连接轨道按状态变色：绿色代表正在接力，橙色代表暂停或确认中，弱化色代表未连接。
- 能力卡顶部必须有状态色条，避免看起来像普通设置卡。
- 已连接后的控制区不再使用整块绿色或橙色容器，避免和主视觉抢焦点；运行/暂停只通过图标、状态胶囊和主按钮表达。
- 实机截图显示首版视觉过重后，默认密度应收敛：标题 24sp 左右，hero 保持可识别但不能像宣传海报。
- 状态胶囊必须使用产品语义色，不出现模板紫色。
- 辅助设置面板要弱于主状态和接力控制，避免和核心任务抢焦点。
- Android 接力页只允许一个强视觉中心：设备桥接 hero。能力卡、接力控制、本机名称、手动连接都必须降级为操作面板。
- “暂停接力”在运行中不是主要增长动作，应使用较轻的 tonal 按钮；“恢复接力”才使用更强的主按钮。
- 下一步动作必须独立成提示卡：通知访问缺失、等待 Mac 确认、已选地址但未请求确认，不应该只藏在能力小卡或手动表单里。
- 已配对状态下默认隐藏手动地址入口；用户需要换 Mac 时走“忘记这台 Mac”再重新发现，避免日常首页退化成配置页。
- 底部主导航保持三项：接力、文件、设备。可靠性、手动连接和诊断是二级入口。

## Main Window IA

主窗口是动作处理台，不是系统监控面板。阅读顺序固定为：

1. 顶部状态
   - 当前连接
   - Android 接力运行 / 暂停状态
   - 今日接收
   - 最近接收
   - 本地接收器状态

2. 主工作区左侧：动作处理
   - 待处理动作优先展示
   - 验证码、打开链接必须比历史记录更醒目
   - 最近文件投递紧跟待处理动作，作为第二高频处理对象
   - 文件投递卡只展示保存结果和文件动作，不混入普通通知列表

3. 主工作区右侧：辅助上下文
   - 进行中状态只在确实存在时出现，包括正在接收文件
   - 最近活动
   - 只用于回看，不抢待处理动作的视觉中心

4. 支持区：诊断与连接详情
   - 放在主工作区之后
   - 默认折叠
   - 异常时显示弱提醒
   - Host、Port、Pairing Token、错误信息放这里

## Menu Bar IA

菜单栏是轻量控制中心：

- 顶部显示连接状态和最近接收时间
- 待处理动作优先
- 进行中状态其次
- 最近三条记录放后面
- 主窗口、设置、退出作为底部操作

不在菜单栏显示：

- Pairing Token
- Host / Port
- 模拟状态按钮
- 过长通知历史

## Visual Rules

- 安静、紧凑、可扫读
- 使用 macOS 原生控件和 SF Symbols
- 卡片只用于具体通知、动作和状态项
- 页面分区保持轻，不做大块灰色堆叠
- 包名不直接作为用户主标题，尽量映射为应用名

## macOS Native Polish

参考外部 macOS design guidelines skill 后，当前 Mac 端 UI 优先遵守这些方向：

- 高频命令进入 App 菜单和窗口 toolbar，而不是只藏在内容区按钮里
- 设置窗口应可缩放，并记住窗口位置
- 设置页优先展示概览、隐私、文件投递和状态卡片；长规则列表默认折叠，避免设置页被规则管理淹没
- 通知卡片支持右键菜单，和省略号菜单保持同一套规则动作
- 卡片 hover 反馈要轻，只表达“可操作”，不制造新的视觉中心
- 图标按钮必须有 tooltip 和 accessibility label
- 诊断、token、host、port 等开发信息默认保持折叠

## Anti-Patterns

- 首页显示大段诊断字段
- 把 `com.tencent.mm` 当作主标题
- 把“临时动作”暴露为用户概念
- 所有通知都用同等重量展示
- 动作反馈远离动作来源
- 菜单栏塞满开发按钮
